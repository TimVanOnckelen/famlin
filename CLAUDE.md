# CLAUDE.md — Famlin

**Famlin** is a private, self-hosted family updates app (posts, photos, comments, likes, milestones). A monorepo with these parts:

- **`backend/`** — Fastify + TypeScript + Prisma (Postgres) API. ESM (`"type": "module"`).
- **`backend/admin/`** — React + Vite admin UI, built into `backend/dist/admin/` and served by the backend at `/admin`.
- **`mobile/`** — Expo / React Native app (iOS + Android).
- **`web/`** — `famlin-web`, a desktop-focused React + Vite member-facing web app (separate from the Expo app — not an RN-web build), consuming `@famlin/api-client` as an npm workspace member. Built into `backend/dist/web/` (vite `outDir`) and served by the backend at `/` from the same container — the exact single-container pattern `backend/admin` uses. Has the design-guide login flow (password + OIDC PKCE, `LoginPage.tsx`) and the core member features: a feed with a multi-select family filter (`FeedPage.tsx` — chips in the feed column, empty selection = all the user's groups, mirroring `GET /api/posts`' `groupIds` semantics; mobile's `FeedScreen` uses the same filter model), post cards with photo hero/lightbox and milestone variants, reactions, favorites, inline comments, a new-post composer with photo upload + a linked-album picker spanning every media source (`NewPostModal.tsx`/`MediaPickerModal.tsx`, backed by the provider-generic `/api/media` endpoints — mobile's `MediaPickerModal` mirrors it), and a profile page mirroring mobile's `ProfileScreen` (`ProfilePage.tsx` — avatar upload, notification-preference toggles, language picker, server info, logout; reached from the header user menu). No client-side routing yet (react-router-dom is a dependency but unused — the profile page is a `view` state switch in `App.tsx`, reset to the feed when the session ends). Design tokens in `web/src/index.css` come from `design/Famlin Styleguide.dc.html` (the source of truth).
- **`packages/api-client/`** — `@famlin/api-client`, the shared axios/data layer (auth, media albums (`media.ts`, provider-generic; `immich.ts` is its legacy Immich-only equivalent), invites, uploads, posts, comments, groups, notifications, push tokens, domain types, react-query cache helpers, and the browser OIDC PKCE login flow in `oidcBrowser.ts` — browser-only, used by `web/`'s login page) consumed by both `mobile/` (via a `file:` dependency) and `web/` (via the npm workspace); a storage-adapter seam (`setStorageAdapter`) lets each consumer plug in its own token persistence (SecureStore/AsyncStorage on mobile, `localStorage` on web) instead of the package depending on any platform API directly. Its barrel (`src/index.ts`) deliberately uses named re-exports rather than `export *` — `export *` compiles to a runtime `__exportStar` call whose re-exported names Rollup can't statically resolve, which breaks `web`'s production `vite build` (works fine under `tsc --noEmit`/Jest/Metro, so this only surfaces at bundle time).
- **`docs/`** — Docusaurus documentation site (deployed separately, not served by the backend).
- **`website/`** — static marketing landing page (plain HTML/CSS, no build step), deployed as the root of GitHub Pages at `famlin.app`.

`website/` and `docs/` deploy together to GitHub Pages via `.github/workflows/pages.yml`: the landing page is copied to the artifact root and the Docusaurus build (`docs/build`) is nested under `/docs`, so `docs.baseUrl` in `docs/docusaurus.config.ts` must stay `/docs/` (not `/`) for internal doc links to resolve correctly.

> The original step-by-step build brief describes how Famlin was meant to be built from scratch (historical intent, not current state) — see [Original project brief](/developers/project-brief) in the Developers tab.

## Commands

Run from the repo root unless noted. `backend/`, `backend/admin/`, `mobile/`, and `docs/` still install independently (`cd backend && npm install`, `cd backend/admin && npm install`, `cd mobile && npm install`, `cd docs && npm install`) — there's no root install for those. `packages/*` and `web/` are an npm workspace instead: a plain root `npm install` builds/links `packages/api-client` (its `prepare` script runs `tsc` automatically) and links `web/`'s `@famlin/api-client` dependency. `mobile/` depends on the package via `file:../packages/api-client`, not the workspace protocol, specifically so it can keep installing independently — after changing `packages/api-client/src/`, run `npm run build --workspace=packages/api-client` (or root `npm install` again) and commit the resulting `packages/api-client/dist/` files before mobile or web pick up the change. The `dist/` output is tracked so that EAS/cloud builds have a usable package even though `file:` dependencies don't get their devDependencies installed before `prepare` runs.

**Backend** (`cd backend`)
- `npm run dev` — `nodemon` (config in `nodemon.json`), watching `src/`, `prisma/schema.prisma`, and `prisma/migrations/`. On every change it kills the previous process and reruns `db:deploy && db:generate && tsx src/server.ts` in that order, so a schema/migration change (a teammate's migration landing via `git pull`, or a migration added while the container was already running) applies to the dev DB and regenerates the Prisma Client automatically — no more manual `prisma generate` + container restart. Tolerates JSON imports.
- `npm run build` — `tsc` typecheck/emit **and** builds the admin UI. Use `npx tsc --noEmit` for a quick typecheck.
- `npm start` — run the compiled server (`node dist/server.js`).
- `npm run db:migrate` — Prisma migrate (dev) · `db:deploy` (prod) · `db:generate` · `db:seed` · `db:studio`.
- `npm test` — Vitest against a dedicated `<db>_test` database (never the real one — `tests/setup/assert-test-database.ts` refuses to run unless `DATABASE_URL`'s database name contains "test", and `tests/setup/test-setup.ts` truncates every table once per test file). Since `docker-compose.yml` doesn't publish Postgres to the host, run `npm run test:docker` (execs into the running `famlin-backend` dev container, which reaches `famlin-db` over the compose network, and creates the test database on first run if needed) rather than plain `npm test` unless you have direct Postgres access. Fixtures shared across a file's tests via a top-level `beforeAll` (e.g. `src/__tests__/authorization.test.ts`) are fine — truncation only happens between files, not between tests in the same file.

**Admin UI** (`cd backend/admin`)
- `npm run dev` — Vite dev server · `npm run build` — production build · `npx tsc --noEmit` — typecheck.

**Mobile** (`cd mobile`)
- `npm start` (Expo) · `npm run ios` · `npm run android` · `npx tsc --noEmit` — typecheck.
- `npm test` — Jest (`jest-expo` preset) unit/hook tests, no device/simulator needed.
- `npm run build:ios` / `build:android` — EAS builds.

**Docs site** (`cd docs`)
- `npm start` — Docusaurus dev server · `npm run build` — static build to `docs/build/` (also regenerates the API reference first) · `npm run typecheck`.
- `npm run gen-api-docs` / `clean-api-docs` — (re)generate `developers/api-reference/` from `docs/openapi/famlin.yaml` (docusaurus-plugin-openapi-docs). The generated MDX + `sidebar.ts` are committed (sidebars-developers.ts imports the sidebar at config-load time, so a fresh checkout must have them), but never hand-edited — edit the spec and regenerate. `npm run build` runs clean+gen automatically; a bare `gen-api-docs` won't overwrite existing files, so run `clean-api-docs` first when regenerating manually.

**Web** (`cd web`, or `npm run dev:web`/`build:web`/`watch:web` from the root)
- `npm run dev` — Vite dev server (port 5174, proxies `/api` to `http://localhost:3000`) · `npm run build` — `tsc && vite build` to `web/dist/` · `npx tsc --noEmit` — typecheck.
- `npm run watch` — `vite build --watch` into `backend/dist/web` for the single-container dev flow (see the `backend/Dockerfile` bullet below); auto-refreshes the browser via a `LIVERELOAD`-gated `livereload` plugin in `vite.config.ts`.
- `npm test` — Vitest + Testing Library (jsdom); shared render helper/fixtures in `src/test/fixtures.tsx`, i18n forced to `en` in `src/test/setup.ts`.

**Shared API client** (`cd packages/api-client`, or `npm run build:api-client` from the root)
- `npm run build` — `tsc` to `dist/` · `npm test` — Vitest · `npx tsc --noEmit` — typecheck.

**Root shortcuts** (`package.json`): `dev:backend`, `dev:admin`, `dev:mobile`, `dev:docs`, `dev:web`, `build:admin`, `build:docs`, `build:web`, `build:api-client`, `db:migrate`, `db:seed`.

After any change, verify with `npx tsc --noEmit` in each affected package.

## Architecture as-built

**Backend routes** (`backend/src/routes/`, registered in `src/server.ts`):

| File | Prefix | Notes |
| --- | --- | --- |
| `auth.ts` | `/api/auth` | Generic OIDC (`/oidc`, `/oidc-config`) + local password login, `/me`, register/reset (admin), first-run admin bootstrap (`/setup-status`, `/setup`), public `/server-info` (`{ version }` from the backend's `package.json`, shown in the mobile app's and web app's profile screens). `/oidc/exchange` (admin), `/oidc/mobile-callback` + `/oidc/mobile-handoff` (mobile) are a server-mediated alternative for providers that require a client secret (Google) — see the OIDC bullet below |
| `groups.ts` | `/api/groups` | **Read-only, member-facing**: your groups, group detail, member list |
| `admin.ts` | `/api/admin` | **All admin mutations**: users, groups CRUD, membership, settings, cross-group content moderation (`/content/posts`, `/content/comments` take optional `groupId`/`authorId`/`q` filters), media album linking (guarded by `requireAdmin`). Media/album mutations: `PATCH /media-albums/:id` `{newAssetMode}` (OFF/MANUAL/AUTO); `GET /media/:provider/people` lists recognized people (`[{id, name, thumbnailDataUri}]`, 400 if provider lacks support); `GET /media/people-links` (rows include `user:{id,name,email}|null`); `POST /media/people-links` `{provider, externalPersonId, label, userId?}` (upsert); `DELETE /media/people-links/:id` |
| `posts.ts` | `/api/posts` | CRUD, always filtered by group membership. `GET /` is a filter over the caller's groups: `?groupIds=` (comma-separated subset), legacy `?groupId=` (one), or no param = every group they belong to; a requested group they're not in is a 403, and a user in no groups gets an empty page. Also `GET /search?groupId=&q=` (case-insensitive match on `content`/`milestoneTag`, still single-group) and `GET /on-this-day?groupId=` (posts created on today's month/day in a prior year) — both registered before `GET /:id` so the literal path segments aren't swallowed by the `:id` param route |
| `comments.ts`, `likes.ts`, `favorites.ts` | `/api` | `favorites.ts` adds `POST /posts/:postId/favorite` (toggle) and `GET /favorites` (the logged-in user's favorited posts, across all their groups). `GET /posts/:postId/comments` takes an optional `?assetUrl=` filter and `POST` an optional `assetUrl` body field to pin a comment to one photo/video (`Comment.assetUrl`) instead of the post as a whole; replies inherit the parent's `assetUrl`. `POST` also takes an optional `attachmentUrl` — a photo/video the commenter uploaded via `POST /api/uploads` as part of the comment itself (`Comment.attachmentUrl`), distinct from `assetUrl` — and `content` is then optional: the two are independently optional, but at least one is required, so a comment can be photo/video-only. `POST` also takes an optional `mentionedUserIds` array — the client resolves "@name" against the group member list, and the server re-validates each id is a current member of the post's group before notifying them (`mention` instead of `new_comment`, so they aren't double-notified). `likes.ts`'s `POST /posts/:postId/like` and `POST /comments/:commentId/like` take an optional `{ type }` body (`ReactionType`: `LIKE`/`LOVE`/`HAHA`/`WOW`/`SAD`/`CARE`, default `LIKE`) — tapping the same reaction again removes it, tapping a different one switches it (still one reaction row per user per post/comment) |
| `push-tokens.ts` | `/api/push-tokens` | `POST` registers, `DELETE` unregisters (called on mobile logout) |
| `api-tokens.ts` | `/api/api-tokens` | Self-service developer **personal access tokens** (`famlin_pat_...` bearer credentials for members' own scripts/integrations — documented for end users in `docs/developers/api.md`): `GET` lists the caller's tokens (never the secret), `POST` creates (plaintext returned exactly once; **session-authenticated requests only** — a PAT gets a 403, so a leaked token can't mint replacements and outlive revocation), `DELETE /:id` revokes. Verification lives in `services/apiTokens.ts` (SHA-256 hash lookup + expiry check) and is wired into `plugins/auth.ts`: `authenticate` branches on the `famlin_pat_` prefix and sets `request.authMethod` (`'session'` \| `'api-token'`), and `authenticateMediaRequest` accepts PATs for `/uploads/*`/Immich media. PATs are deliberately independent of `User.tokenVersion` (password reset keeps integrations running; revocation = row deletion, user deletion cascades). Managed from the web app's avatar menu (`web/src/components/ApiTokensModal.tsx`) — **web only by design, no mobile screen** |
| `notifications.ts` | `/api/notifications` | in-app notification history |
| `uploads.ts` | `/api/uploads` | direct photo upload to a Docker volume, served at `/uploads/` (auth-gated, see below); `GET /media-token` issues the token clients use to read it |
| `media.ts` | `/api/media` | **Read-only, member-facing, provider-generic** (backed by the `MediaProvider` registry, `services/media/registry.ts` — `immich` + `local` today): `GET /groups/:groupId/albums` (a group's linked albums across every media source, each carrying its `provider`; a link whose source is failing reports `assetCount: 0` rather than sinking the whole list), `GET /albums/:linkId/assets` (asset list with optional `?personId=` filter to show only assets of a mapped person; 404 if person isn't mapped, 400 if provider can't filter by person), `GET /groups/:groupId/photos` (merged photo timeline across linked albums + direct post uploads, cursor-paginated with optional `?personId=` filter; 403 non-member, 404 unmapped personId; backed by `services/media/photoTimeline.ts` + `services/media/personFilter.ts`), `GET /assets/:linkId/:assetId/:variantExt` (streams a rendition from whichever provider the link belongs to; `thumbnail`/`preview` are always `.jpg`, `original` matches the real asset type — see `parseMediaAssetPath()` in `types.ts`; `streamAsset()` implementations honor the client's `Range` header and produce/relay `206`/`Content-Range`, which native video players require to play mp4s at all — don't strip that, it breaks video on mobile only, since web `<video>` tolerates a full-body `200`). Gated by `authenticateMediaRequest` (`plugins/auth.ts`), the same dual session-token-or-media-token check `/uploads/*` uses, since it's loaded as an `<Image>`/`<Video>` `uri` — and by the provider's `isAssetInAlbum()` (a required part of the `MediaProvider` contract, not an optimization; Immich caches it 60s, local stats the file), which re-checks the requested asset id actually belongs to the linked album rather than trusting the linkId→group check alone. Linking/unlinking albums is an admin mutation in `admin.ts` (`/api/admin/groups/:id/media-albums`, `/api/admin/media-albums/:id`, per-provider catalog `/api/admin/media/:provider/albums`, connection tests `/api/admin/immich/test` + `/api/admin/media/local/test`). New: `GET /api/media/people?groupId=` (lists people mapped to this group: `[{id, provider, label, userId?}]`, 403 if non-member) — admin mutations for people mapping are in `admin.ts` |
| `immich.ts` | `/api/immich` | **Legacy compat, Immich-only**: the same three member endpoints as `media.ts` but scoped to `provider: 'immich'` links and generating `/api/immich/assets/...` URLs. Kept because that URL shape is stored in existing posts' `uploadedAssetUrls` and still generated by pre-built mobile clients — new client code should use `/api/media`. Both files delegate to the same `immichProvider` (`services/media/immich.ts`) |
| `invites.ts` | `/api/invites` | **Public**: invite preview (`GET /:token`), self-service registration (`POST /:token/register`), join-for-logged-in-user (`POST /:token/accept`, authenticated). Creation/list/revoke are admin mutations in `admin.ts` instead (`/api/admin/groups/:id/invites`, `/api/admin/invites/:id`) |
| `invite-landing.ts` | `/invite/:token` (no prefix) | **Public**: server-rendered HTML the invite link opens; hands off to the app via `famlin://invite/:token?server=...`. Matches the app's design (logo, teal palette) and is translated into the server's `defaultLanguage` setting (no per-visitor Accept-Language negotiation, since there's no signed-in user yet) |
| `landing.ts` | `/` (no prefix) | **Fallback only**: server-rendered HTML landing page (logo + welcome message, same shell as the invite landing page via `utils/html-page.ts`), registered only when `dist/web` doesn't exist. Normally the `web/` SPA is served at `/` instead, with an SPA `index.html` not-found fallback for GET routes outside `/api/*`, `/uploads/*`, and `/admin*` |

Services live in `backend/src/services/` (`settings.ts`, `notifications.ts`, `invites.ts`, `users.ts`, `groups.ts`, `posts.ts`, `comments.ts`, `reactions.ts`, `onThisDay.ts`, `pagination.ts`, `apiTokens.ts`, plus two extension-point directories: `media/` — the `MediaProvider` interface/registry with the `immich` and `local` providers, see the `media.ts` route row above — and `notificationChannels/` — the `NotificationChannel` interface with the `push` and `email` channels, see the notifications bullet in Conventions). `src/events.ts` is the typed in-process **domain event bus** (`post.created`, `comment.created`, `reaction.added`): routes emit facts after their write commits; subscribers in `src/subscribers/` (registered idempotently by `buildApp()`, so tests get the same wiring) decide what the facts mean. Every handler runs fire-and-forget and error-trapped — a broken subscriber can never fail or slow the emitting request (pinned by `tests/events.test.ts`). The notifications subscriber (`src/subscribers/notifications.ts`) owns all "who gets told what" policy (thread participants, mention validation, reaction targets) — routes know nothing about notifications; don't re-add direct `notifyGroup`/`notifyUsers` calls to routes. The admin SPA is served at `/admin` with `index.html` fallback for client-side routing. `src/jobs/onThisDay.ts` runs once a day and `src/jobs/newAssets.ts` runs hourly (minute 15, both via `node-cron`, scheduled in `server.ts` — not `app.ts`, so the test suite building the app via `buildApp()` never triggers jobs): onThisDay notifies each group about posts made on that day in a prior year, while newAssets diffs linked albums against their provider and either notifies the group (MANUAL mode) or auto-creates a post (AUTO mode) when new assets appear.

**List endpoints are cursor-paginated.** `GET /api/posts`, `GET /api/posts/search`, `GET /api/favorites`, and the admin `/users`/`/content/posts`/`/content/comments` endpoints take `?cursor=`/`?take=` (default 30, max 100) and return `{ items, nextCursor }` instead of a bare array — use `services/pagination.ts`'s `paginationArgs`/`paginate` for any new list endpoint rather than returning everything unbounded. `GET /api/posts/on-this-day` doesn't paginate (expected result set is small: at most a handful of years' worth of same-day posts).

**Data model** (`backend/prisma/schema.prisma`): `User`, `Setting`, `Group`, `GroupMember` (M2M join, `@@unique([groupId, userId])`), `Post` (belongs to exactly one `Group`), `Comment` (`attachmentUrl` optionally holds a photo/video the commenter uploaded as part of the comment itself, distinct from `assetUrl`'s pin to an existing post asset), `Like` (a reaction — `type: ReactionType` is `LIKE`/`LOVE`/`HAHA`/`WOW`/`SAD`/`CARE`, one row per user per post/comment via `@@unique([postId, userId])`/`@@unique([commentId, userId])`; post responses also expose `recentReactors` — the three newest reactors' `{ id, name, avatarUrl }`, via `shapePost` in `services/posts.ts` — for the mobile app's overlapping reaction avatars); post responses also carry `people` — an array of admin-mapped media people recognized in that post's photos, computed read-time in `services/media/personTags.ts` by querying each linked album's asset-to-people index (Immich metadata search + per-asset fallback, 10-minute cache) and merging tagged assets with the same person-label into one chip, with fail-soft semantics (provider outage → empty array, never a failed request). `Favorite` (a user's bookmarked posts, `@@unique([postId, userId])`, mirrors `Like`), `PushToken`, `ApiToken` (a developer personal access token — stores only a SHA-256 `tokenHash` of the `famlin_pat_...` secret plus a display `tokenPreview`, optional `expiresAt`, throttled `lastUsedAt`; see `api-tokens.ts` above), `Notification`, `Invite`, `MediaAlbumLink` (links one external media album to one group: `provider` is the registry id — `'immich'`/`'local'`, never rename these once deployments have linked albums — and `externalAlbumId` is that provider's album identifier (an Immich uuid, a local folder name), `@@unique([groupId, provider, externalAlbumId])`; `newAssetMode` controls how new assets on this album are surfaced (`OFF` = never, `MANUAL` = a notification reusing the new-post preference columns, `AUTO` = an auto-created post); `newAssetsCheckedAt` is the last time the new-assets job ran for this link, used to diff against the provider. The proxy routes additionally re-check the requested asset actually belongs to that album, see `media.ts` above. Born as `ImmichAlbumLink` — renamed + backfilled by the `20260709100000_generalize_media_album_link` migration), `MediaPersonLink` (maps a provider's face/person identity to a display label and optional user: `provider`/`externalPersonId` unique, `userId` nullable; only admin-mapped people are visible to members; enables per-person asset filtering). `Post`, `Comment`, and `User` are all hard-deleted — `DELETE /api/posts/:id`, `DELETE /api/comments/:id`, and `DELETE /api/admin/users/:id` permanently remove the row via `prisma.*.delete()`, with cascades doing the rest (deleting a post removes its comments/likes/favorites/notifications; deleting a user removes their posts/comments/likes/favorites/etc. too). There is no restore. `User.tokenVersion` is embedded in every JWT and checked on each request, so bumping it (password change/reset) invalidates every previously-issued token immediately. `Notification.type` is a plain string (not a DB enum) — the closed set of values (including `mention` and `on_this_day`) lives in `NotifyType` in `services/notificationChannels/types.ts` (re-exported from `services/notifications.ts`); both currently reuse the `emailOnNewComment`/`pushOnNewComment` and `emailOnNewPost`/`pushOnNewPost` preference columns respectively rather than adding dedicated ones.

**Mobile** (`mobile/src/`): screens in `screens/`, navigation in `navigation/` + `App.tsx`, Zustand store in `stores/`, API layer in `api/` (axios; base URL is user-entered at login, not hardcoded), React Query for fetching/caching.

## Conventions & gotchas

- **Group membership is the core authorization rule.** Every posts query must filter on groups the logged-in user belongs to — a user must never see posts from a group they aren't a member of. Group/member mutations require `isAdmin`.
- **Group mutations live only in `admin.ts`.** `groups.ts` is deliberately read-only — do not re-add create/update/delete or member-management routes there.
- **`requireAdmin` returns a boolean and sends the 403 itself.** Callers **must** `return` when it returns `true`, otherwise the handler keeps running after the response is sent.
- **Runtime config is in the DB, not `.env`.** OIDC issuer/client ID/scopes, allowed emails, SMTP, Immich server URL/API key, the local-media root path (`localMediaPath`), and the server's `defaultLanguage`/`appStoreUrl`/`playStoreUrl` live in the `Setting` table (managed via the admin UI's "General"/"Media integrations" sections), read through `getAllSettings()` (`services/settings.ts`), which caches for 10s and is invalidated on write. Each media source is one server-level connection shared by every group — `getImmichSettings()`/`getLocalMediaSettings()` report whether it's `configured`; which albums are visible to which group is a separate per-group `MediaAlbumLink` row, not part of `Setting`. `defaultLanguage` is used both for server-rendered pages with no signed-in user (the `/invite/:token` landing page) and for notification push/email text, since recipients don't have a stored per-user language preference — see `getT`/`SUPPORTED_LANGUAGES` in `src/i18n/index.ts`. `appStoreUrl`/`playStoreUrl` are optional; when set (or defaulted), the invite landing page shows download buttons. `appStoreUrl` is blank by default (no single iOS listing — each deployment distributes its own build). `playStoreUrl` defaults to the official pre-built Android app (`https://play.google.com/store/apps/details?id=be.xeweb.famlin`, package `be.xeweb.famlin` per `mobile/app.json`) whenever it's never been explicitly saved — an admin can still clear it to hide the button, which is preserved as a real blank rather than falling back to the default (see the `map.has()` check in `loadAllSettings()`, `services/settings.ts`). Only `DATABASE_URL`, `PORT`, `NODE_ENV`, `JWT_SECRET`, and `TRUST_PROXY` are startup env vars — `TRUST_PROXY` (default `false`) must be `true` when (and only when) Famlin sits behind a reverse proxy that sets `X-Forwarded-Proto`/`X-Forwarded-Host`, otherwise those headers are ignored so a directly-exposed server can't have its origin spoofed.
- **OIDC login works with any provider** (Google, Microsoft, Authentik, Keycloak, Auth0, ...) via standard discovery + JWKS verification (`jose`, see `plugins/auth.ts` `verifyOidcToken`/`getDiscovery`). By default, mobile and admin both do a public-client Authorization Code + PKCE flow directly against the provider (no client secret) and hand the resulting ID token to `POST /api/auth/oidc`; `GET /api/auth/oidc-config` is the public endpoint clients use to discover the issuer's endpoints (plus `usesClientSecret`/`mobileCallbackUrl`, see below). It respects `allowedEmails` — new accounts are only provisioned if the email is whitelisted (empty list = allow all); see `services/settings.ts` `isEmailAllowed`. All three OIDC entry points share one user-provisioning/token-issuing function, `completeOidcLogin()` in `services/oidcLogin.ts`.
- **Google needs the optional `oidcClientSecret` setting, not the plain PKCE flow above.** Google's token endpoint rejects a secretless exchange, and its native/"iOS"/"Android" OAuth client types force `redirect_uri` onto a Google-generated `com.googleusercontent.apps.<id>` scheme rather than the app's own `famlin://` — a hard blocker since Famlin ships one shared pre-built Android app (`be.xeweb.famlin`) across many independent deployments, each with a different Google client and therefore a different mandated scheme that can't be baked into that shared binary. Setting `oidcClientSecret` switches both surfaces to a server-mediated exchange instead, so Google can be configured as one ordinary **Web application** client: the admin UI keeps its existing `https://.../admin/` redirect but POSTs the code to `POST /api/auth/oidc/exchange` instead of exchanging it itself; the mobile app authorizes against `mobileCallbackUrl` (an HTTPS URL under the deployment's own domain, from `GET /oidc-config`), and `GET /api/auth/oidc/mobile-callback` finishes the exchange server-side and 302s to `famlin://oidc-callback?handoff=<code>` — a real redirect_uri scheme change would break the deep link contract, so the token itself is never put in that URL's query string (would leak into access logs); the app redeems the one-time `handoff` code via `POST /api/auth/oidc/mobile-handoff` (`services/oidcHandoff.ts`, in-memory, single-use, 2-minute TTL — see `utils/oidcLogin.ts` in the mobile app).
- **i18n is mandatory — no hardcoded UI strings.** Use `const { t } = useTranslation()` and keys. Locales: `mobile/src/i18n/locales/{en,nl}.json` and `backend/admin/src/i18n/locales/{en,nl}.json`; `en.json` is the source, `nl.json` the translation. Backend error messages use `getT(request)('errors.…')`. Adding a UI string means adding both `en` and `nl` keys. Weblate component type: **i18next JSON**; `fallbackLng` is `en`.
- **Backend is ESM + `NodeNext`.** JSON imports need the attribute: `import x from './x.json' with { type: 'json' };`.
- **Removing a group member keeps their existing posts/comments visible** to the remaining members (deliberate MVP choice).
- **Posts, comments, and users are all hard-deleted — there is no soft-delete anywhere.** `DELETE /api/posts/:id`/`DELETE /api/comments/:id` call `prisma.post.delete`/`prisma.comment.delete` directly, and `DELETE /api/admin/users/:id` calls `prisma.user.delete`; nothing filters on a `deletedAt` column because that column doesn't exist. Deleting a user cascades to remove their posts, comments, likes, favorites, and notifications immediately (`onDelete: Cascade` on `Post.author`/`Comment.author` etc. in `schema.prisma`) — this is intentional, don't add back a `deletedAt` column or a restore endpoint to soften it. The last remaining admin can't be deleted or demoted (`isLastAdmin` in `routes/admin.ts`) — keep that check if you touch those routes. `backend/src/__tests__/authorization.test.ts` has regression tests confirming a deleted post/comment/user is fully gone (404s/401s, not just hidden).
- **There is no default/seeded admin account.** A fresh install has zero users; `GET /api/auth/setup-status` reports `needsSetup: true` until the first user exists, and the admin UI shows a one-time setup screen (`SetupPage.tsx`) instead of the login form in that state. `POST /api/auth/setup` provisions that first account as an admin and is guarded by a Postgres advisory lock (`SETUP_ADVISORY_LOCK_KEY` in `routes/auth.ts`) so two concurrent requests can't both pass the "no users yet" check — it 409s once a user exists. `prisma/seed.ts` (`db:seed`) is a **local dev/test fixture script only** (fake family + `admin@example.com`/`test123456`), never run automatically by the Docker image or `prisma migrate deploy` — don't treat it as production onboarding.
- **Invite links are their own authorization and deliberately bypass `allowedEmails`.** A valid, unexpired, unused `Invite` (generated per-group from the admin UI) lets `POST /api/auth/oidc` / `POST /api/auth/login` (via an optional `inviteToken` field) and `POST /api/invites/:token/register` provision or join a user even when the email isn't whitelisted — see `verifyOidcToken(idToken, { allowUnlisted })` in `plugins/auth.ts` and `consumeInvite`/`getValidInvite` in `services/invites.ts`. Don't add new email-allowlist checks upstream of these paths without accounting for the invite override. `consumeInvite` runs as a transaction with an atomic conditional update so two concurrent redemptions of the same single-use invite can't both succeed.
- **`/uploads/*` requires auth — session token or media token.** See [Photos and uploads](docs/developers/architecture.md#photos-and-uploads) in the architecture doc. `uploadedAssetUrls`/`assetUrl` are validated (zod regex in `types.ts`) against the exact path shape the upload route writes — don't loosen that to a bare `z.string()`, it's what stops a post from linking to an arbitrary external URL.
- **Don't leak `err.message` to the client.** The global error handler (`app.ts`) turns a `ZodError` into a translated `400` and anything else into a generic translated `500`; a route's own `catch` block should only special-case errors it needs to give a specific, translated, non-sensitive message for (see the `OidcError`/`P2025` handling in `routes/auth.ts` as the pattern) — let everything else propagate to the global handler instead of echoing `err.message`.
- **Mark breaking changes in commit messages — release notes depend on it.** Commits follow Conventional Commits, and any change that would require action from an existing deployment on upgrade (incompatible API/response change, newly required env var, behavior-altering default) must carry a `!` after the type (`feat!: ...`) or a `BREAKING CHANGE: <what an admin must do>` footer. release-please builds the release's "⚠ BREAKING CHANGES" changelog section and the version bump from that marker, and the AI release-notes rewrite in `release-please.yml` only warns admins about marked commits — an unmarked breaking change ships silently. The `api-breaking-changes` job in `ci.yml` enforces the API half: it runs oasdiff on `docs/openapi/famlin.yaml` and fails a PR whose spec change is breaking with no marked commit/PR title; non-API breakage (env vars, deploy steps) relies on the commit author — that includes you — marking it.

## Documentation

`docs/` (Docusaurus) has two tabs — **Docs** (`docs/docs/`, self-hosting/ops) and **Developers** (`docs/developers/`, working on the code). When a change lands, check whether it needs a matching doc update:

| Change | Doc to update |
| --- | --- |
| New/changed backend route, prefix, or data model field | `docs/developers/architecture.md` (routes table / data model section) — and the table in this file's [Architecture as-built](#architecture-as-built) section above |
| New/changed member-facing endpoint, request/response field, auth mechanism, pagination/error convention, or rate limit | `docs/openapi/famlin.yaml` — the hand-maintained OpenAPI spec the generated API reference (`docs/developers/api-reference/`, docusaurus-plugin-openapi-docs) is built from; regenerate with `npm run gen-api-docs` after editing (clean first, see Docs commands). Auth/conventions/recipes prose lives in `docs/developers/api.md` |
| New/changed startup env var, Docker Compose service, or deploy step | `docs/docs/server-setup.md` |
| New/changed OIDC/SSO, allowed-emails, or SMTP setup step | `docs/docs/admin-configuration.md` |
| New/changed invite flow (creation, redemption, expiry/revocation) | `docs/docs/inviting-family.md` |
| New/changed admin capability for users, groups, or content moderation | `docs/docs/managing-users-and-content.md` |
| New/changed local dev command, seed behavior, or Docker dev workflow | `docs/developers/quick-start.md` |
| New contribution requirement or PR process change | `docs/developers/contributing.md` |
| Security-relevant change (auth, secrets handling, permissions) | `docs/docs/security.md` |
| New/changed marketing claim, install steps, or doc link on the landing page | `website/index.html` |

Not every change needs a doc update — skip it for internal refactors with no user- or developer-visible behavior change. When in doubt, err toward updating: stale docs are worse than a slightly redundant edit.

## Deploy

Docker Compose (`docker-compose.yml`): `famlin-db` (Postgres, persistent volume, **not** published to the host — reached only over the compose network) + `famlin-backend` (Node 20, runs as the non-root `node` user), pulled by default as the pre-built `ghcr.io/timvanonckelen/famlin:${FAMLIN_VERSION:-latest}` image rather than built locally. Requires `JWT_SECRET` and `POSTGRES_PASSWORD` in `.env` (see `.env.example`); `docker compose up` fails fast with a clear error if either is unset. First run: open `/admin` — a fresh database has no users, so it shows a one-time setup screen to create the first (admin) account — then configure OIDC/SSO, allowed emails, and SMTP there. Keep the Postgres volume in the existing NAS backup scope. `docker-compose.override.yml` (local dev only, auto-merged by plain `docker compose up`) is what actually builds from `backend/Dockerfile` for hot-reload — production deploys use `docker-compose.yml` alone and never build from source unless deliberately switched over (see `docs/docs/maintenance.md#building-from-source-instead`).

**`backend/Dockerfile` builds from the repo root, not `backend/`** — the image bundles the member-facing web app (`web/` + `packages/api-client`, both outside `backend/`) in a dedicated `webbuild` stage (root workspace `npm ci` + `vite build` → copied to `dist/web`), alongside the backend + admin build. Both `docker-compose.override.yml` and `docker-publish.yml` therefore use `context: .` with `dockerfile/file: backend/Dockerfile`; the root `.dockerignore` keeps the context lean. In local dev the container doesn't build web itself (the root workspace isn't mounted in) — instead the overlay bind-mounts `./backend/dist/web` through the anonymous `/app/dist` volume, so a **host-side** `npm run build:web` (or `npm run watch:web` for rebuild-on-change) makes the dev container serve the web app at `http://localhost:3000/`; restart the backend after the *first* build so it detects `dist/web/index.html` (`app.ts` checks for the file, not the directory, since the bind mount creates an empty dir). `npm run watch:web` also auto-refreshes the browser tab: it sets `LIVERELOAD=1`, which `web/vite.config.ts`'s `watchLivereload()` plugin uses to start a `livereload` server watching `outDir` and inject its client snippet into the built `index.html` — never set by plain `vite build`, so the real production bundle in the Docker image never carries it. Alternatively `npm run dev:web` runs the Vite dev server on 5174 with its own HMR, proxying `/api` to the container.

**`.github/workflows/docker-publish.yml`** builds and pushes `backend/Dockerfile` to `ghcr.io/timvanonckelen/famlin`, tagged with the release semver plus a rolling `latest`. It's a **reusable workflow** (`workflow_call` with an optional `tag_name` input) plus a manual `workflow_dispatch` entry point — it is **not** triggered by `on: release`, because release-please creates the release with the default `GITHUB_TOKEN` and GitHub suppresses workflow-triggered-workflow events. Instead, `.github/workflows/release-please.yml` cuts releases from conventional commits on `main` and, in the same run, chains a `publish-docker` job (gated on `release_created`) that calls `docker-publish.yml` with the new tag — so the image build needs no manual trigger once a release lands.

**Website & docs** deploy separately to GitHub Pages via `.github/workflows/pages.yml`, triggered on pushes to `website/**` or `docs/**`: it builds `docs/` with Docusaurus, assembles `website/` (root) + `docs/build` (under `/docs`) into one artifact, and publishes it to `famlin.app` (custom domain via `website/CNAME`).
