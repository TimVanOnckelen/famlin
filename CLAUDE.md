# CLAUDE.md — Famlin

**Famlin** is a private, self-hosted family updates app (posts, photos, comments, likes, milestones). A monorepo with these parts:

- **`backend/`** — Fastify + TypeScript + Prisma (Postgres) API. ESM (`"type": "module"`).
- **`backend/admin/`** — React + Vite admin UI, built into `backend/dist/admin/` and served by the backend at `/admin`.
- **`mobile/`** — Expo / React Native app (iOS + Android).
- **`web/`** — `famlin-web`, a desktop-focused React + Vite member-facing web app (separate from the Expo app — not an RN-web build), consuming `@famlin/api-client` as an npm workspace member. Built into `backend/dist/web/` (vite `outDir`) and served by the backend at `/` from the same container — the exact single-container pattern `backend/admin` uses. Has the design-guide login flow (password + OIDC PKCE, `LoginPage.tsx`) and the core member features: a feed with a multi-select family filter (`FeedPage.tsx` — chips in the feed column, empty selection = all the user's groups, mirroring `GET /api/posts`' `groupIds` semantics; mobile's `FeedScreen` uses the same filter model), post cards with photo hero/lightbox and milestone variants, reactions, favorites, inline comments (a `TRIP` post gets a read-only journal view — trip feed cards, day counter, check-in timeline, photo collage — mirroring mobile's `TripDetailScreen` but deliberately view-only: no trip creation, check-in composing, closing, or traveler management on web; `web/src/utils/trip.ts` carries the same day-number/check-in-splitting logic as mobile's `trip.ts` util), a new-post composer with photo upload + a linked-album picker spanning every media source (`NewPostModal.tsx`/`MediaPickerModal.tsx`, backed by the provider-generic `/api/media` endpoints — mobile's `MediaPickerModal` mirrors it), and a profile page mirroring mobile's `ProfileScreen` (`ProfilePage.tsx` — avatar upload, notification-preference toggles, language picker, server info, logout; reached from the header user menu). No client-side routing yet (react-router-dom is a dependency but unused — the profile page is a `view` state switch in `App.tsx`, reset to the feed when the session ends). Desktop-focused, but on small screens (`BottomNav.css`'s `max-width: 720px`) the four top-level pages (Feed/Photos/Chat/Profile) additionally render `BottomNav.tsx` — a fixed app-style tab bar mirroring mobile's `MainTabs.tsx` (same four tabs, plus a floating "+" FAB on Feed/Photos only, same as mobile's `FAB_TABS`) — while `AppHeader.tsx` hides its now-redundant new-post button and chat icon at that width. Each page owns its own composer state already (for `AppHeader`'s new-post button), so `BottomNav`'s FAB just reuses that same per-page callback rather than composer state living anywhere centrally. Design tokens in `web/src/index.css` come from `design/Famlin Styleguide.dc.html` (the source of truth).
- **`packages/api-client/`** — `@famlin/api-client`, the shared axios/data layer (auth, media albums (`media.ts`, provider-generic; `immich.ts` is its legacy Immich-only equivalent), invites, uploads, posts, comments, groups, notifications, push tokens, domain types, react-query cache helpers, and the browser OIDC PKCE login flow in `oidcBrowser.ts` — browser-only, used by `web/`'s login page) consumed by both `mobile/` (via a `file:` dependency) and `web/` (via the npm workspace); a storage-adapter seam (`setStorageAdapter`) lets each consumer plug in its own token persistence (SecureStore/AsyncStorage on mobile, `localStorage` on web) instead of the package depending on any platform API directly. Its barrel (`src/index.ts`) deliberately uses named re-exports rather than `export *` — `export *` compiles to a runtime `__exportStar` call whose re-exported names Rollup can't statically resolve, which breaks `web`'s production `vite build` (works fine under `tsc --noEmit`/Jest/Metro, so this only surfaces at bundle time).
- **`docs/`** — Docusaurus documentation site (deployed separately, not served by the backend).
- **`website/`** — static marketing landing page (plain HTML/CSS, no build step), deployed as the root of GitHub Pages at `famlin.app`.

`website/` and `docs/` deploy together to GitHub Pages via `.github/workflows/pages.yml`: the landing page is copied to the artifact root and the Docusaurus build (`docs/build`) is nested under `/docs`, so `docs.baseUrl` in `docs/docusaurus.config.ts` must stay `/docs/` (not `/`) for internal doc links to resolve correctly.

> The original step-by-step build brief describes how Famlin was meant to be built from scratch (historical intent, not current state) — see [Original project brief](/developers/project-brief) in the Developers tab.

## Commands

Run from the repo root unless noted. `backend/`, `backend/admin/`, `mobile/`, and `docs/` still install independently (`cd backend && npm install`, `cd backend/admin && npm install`, `cd mobile && npm install`, `cd docs && npm install`) — there's no root install for those. `packages/*` and `web/` are an npm workspace instead: a plain root `npm install` builds/links `packages/api-client` (its `prepare` script runs `tsc` automatically) and links `web/`'s `@famlin/api-client` dependency. `mobile/` depends on the package via `file:../packages/api-client`, not the workspace protocol, specifically so it can keep installing independently — after changing `packages/api-client/src/`, run `npm run build --workspace=packages/api-client` (or root `npm install` again) and commit the resulting `packages/api-client/dist/` files before mobile or web pick up the change. The `dist/` output is tracked so that EAS/cloud builds have a usable package even though `file:` dependencies don't get their devDependencies installed before `prepare` runs.

**Backend** (`cd backend`)
- `npm run dev` — `nodemon` (config in `nodemon.json`), watching `src/`, `prisma/schema.prisma`, and `prisma/migrations/`. On every change it kills the previous process and reruns `db:deploy && db:generate && tsx src/server.ts` in that order, so a schema/migration change (a teammate's migration landing via `git pull`, or a migration added while the container was already running) applies to the dev DB and regenerates the Prisma Client automatically — no more manual `prisma generate` + container restart. Tolerates JSON imports.
- `npm run build` — `tsc` typecheck/emit **and** builds the admin UI. Use `npx tsc --noEmit` for a quick typecheck.
- `npm start` — run the compiled server (`node dist/server.js`).
- `npm run db:migrate` — Prisma migrate (dev) · `db:deploy` (prod) · `db:generate` · `db:seed` · `db:studio`.
- `npm test` — Vitest against a dedicated `<db>_test` database (never the real one — `tests/setup/assert-test-database.ts` refuses to run unless `DATABASE_URL`'s database name contains "test", and `tests/setup/test-setup.ts` truncates every table once per test file). Since `docker-compose.yml` doesn't publish Postgres to the host, run `npm run test:docker` (execs into the running `famlin-backend` dev container, which reaches `famlin-db` over the compose network, and creates the test database on first run if needed) rather than plain `npm test` unless you have direct Postgres access. Fixtures shared across a file's tests via a top-level `beforeAll` (e.g. `src/__tests__/authorization.test.ts`) are fine — truncation only happens between files, not between tests in the same file.

**Admin UI** (`cd backend/admin`)
- `npm run dev` — Vite dev server · `npm run build` — production build · `npx tsc --noEmit` — typecheck.

**Mobile** (`cd mobile`)
- `npm start` (Expo) · `npm run ios` · `npm run android` · `npx tsc --noEmit` — typecheck.
- `npm test` — Jest (`jest-expo` preset) unit/hook tests, no device/simulator needed.
- `npm run build:ios` / `build:android` — EAS builds.

**Docs site** (`cd docs`)
- `npm start` — Docusaurus dev server · `npm run build` — static build to `docs/build/` (also regenerates the API reference first) · `npm run typecheck`.
- `npm run gen-api-docs` / `clean-api-docs` — (re)generate `developers/api-reference/` from `docs/openapi/famlin.yaml` (docusaurus-plugin-openapi-docs). The generated MDX + `sidebar.ts` are committed (sidebars-developers.ts imports the sidebar at config-load time, so a fresh checkout must have them), but never hand-edited — edit the spec and regenerate. `npm run build` runs clean+gen automatically; a bare `gen-api-docs` won't overwrite existing files, so run `clean-api-docs` first when regenerating manually.

**Web** (`cd web`, or `npm run dev:web`/`build:web`/`watch:web` from the root)
- `npm run dev` — Vite dev server (port 5174, proxies `/api` to `http://localhost:3000`) · `npm run build` — `tsc && vite build` to `web/dist/` · `npx tsc --noEmit` — typecheck.
- `npm run watch` — `vite build --watch` into `backend/dist/web` for the single-container dev flow (see the `backend/Dockerfile` bullet below); auto-refreshes the browser via a `LIVERELOAD`-gated `livereload` plugin in `vite.config.ts`.
- `npm test` — Vitest + Testing Library (jsdom); shared render helper/fixtures in `src/test/fixtures.tsx`, i18n forced to `en` in `src/test/setup.ts`.

**Shared API client** (`cd packages/api-client`, or `npm run build:api-client` from the root)
- `npm run build` — `tsc` to `dist/` · `npm test` — Vitest · `npx tsc --noEmit` — typecheck.

**Root shortcuts** (`package.json`): `dev:backend`, `dev:admin`, `dev:mobile`, `dev:docs`, `dev:web`, `build:admin`, `build:docs`, `build:web`, `build:api-client`, `db:migrate`, `db:seed`.

After any change, verify with `npx tsc --noEmit` in each affected package.

## Architecture as-built

**Backend routes** (`backend/src/routes/`, registered in `src/server.ts`):

| File | Prefix | Notes |
| --- | --- | --- |
| `auth.ts` | `/api/auth` | Generic OIDC (`/oidc`, `/oidc-config`) + local password login, `/me`, register/reset (admin; `POST /register` also takes an optional `groupIds: string[]`, validated to exist and assigned atomically with the new user in one transaction), first-run admin bootstrap (`/setup-status`, `/setup`), public `/server-info` (`{ version, minAppVersion, appStoreUrl, playStoreUrl, readOnly }` — `version` is the backend's own `package.json` version, shown in the mobile app's and web app's profile screens; `minAppVersion` is a module-level constant, the oldest mobile app version this server still supports, used by the mobile app to show a blocking "update required" screen — bump it only when compatibility with older clients is genuinely dropped, and mark that commit as breaking; `appStoreUrl`/`playStoreUrl`, from `Setting` (`null` when unset), let that screen link to the right store for this deployment; `readOnly` is `true` when the server is running in demo/read-only mode, so clients can hide or warn about disabled mutations). `/oidc/exchange` (admin), `/oidc/mobile-callback` + `/oidc/mobile-handoff` (mobile) are a server-mediated alternative for providers that require a client secret (Google) — see the OIDC bullet below |
| `groups.ts` | `/api/groups` | **Read-only, member-facing**: your groups, group detail, member list. List/detail responses also carry `allowedPostTypes: string[]` — the group's resolved, effective list of creatable post types (never empty; see the custom-post-types bullet below) |
| `admin.ts` | `/api/admin` | **All admin mutations**: users, groups CRUD, membership, settings, cross-group content moderation (`/content/posts`, `/content/comments` take optional `groupId`/`authorId`/`q` filters), media album linking (guarded by `requireAdmin`). Media/album mutations: `PATCH /media-albums/:id` `{newAssetMode}` (OFF/MANUAL/AUTO); `GET /media/:provider/people` lists recognized people (`[{id, name, thumbnailDataUri}]`, 400 if provider lacks support); `GET /media/people-links` (rows include `user:{id,name,email}|null`); `POST /media/people-links` `{provider, externalPersonId, label, userId?}` (upsert); `DELETE /media/people-links/:id`. `GET /post-types` → `{items:[{id}]}` from the `PostTypeHandler` registry; group create/update bodies take an optional `allowedPostTypes: string[]` (each id validated against the registry, `errors.unknownPostType` if not; omitted on create or `[]` = all types, including ones registered later; omitted on `PATCH` = unchanged), and admin group responses carry the **raw** stored array (not resolved), letting the admin UI distinguish "all" from an explicit list. `POST /content/posts/:id/retrigger-push` manually resends a post's `new_post` push notification (push-only — no email, no duplicate in-app `Notification` row; 400 `errors.pushNotificationsDisabled` if push is off server-wide) via `resendPostPush()` in `services/notifications.ts`; `GET /push-log?postId=` lists `PushDeliveryLog` rows (cursor-paginated, optional post filter) — see the push-delivery-logging bullet below. `GET /export` streams a zip of all family content (users sans credentials, groups/members, posts, comments, reactions, favorites, chat) plus the whole uploads directory (`services/export.ts`, `archiver` in store mode); `Setting`/invites/push/API tokens/notification history are deliberately excluded — this is family content + media, not server config or credentials |
| `posts.ts` | `/api/posts` | CRUD, always filtered by group membership. `GET /` is a filter over the caller's groups: `?groupIds=` (comma-separated subset), legacy `?groupId=` (one), or no param = every group they belong to; a requested group they're not in is a 403, and a user in no groups gets an empty page. Also `GET /search?groupId=&q=` (case-insensitive match on `content`/`milestoneTag`, still single-group) and `GET /on-this-day?groupId=` (posts created on today's month/day in a prior year) — both registered before `GET /:id` so the literal path segments aren't swallowed by the `:id` param route. `POST /` also takes `groupIds` (1–20, alongside legacy `groupId`) for **cross-posting**: one post per target group is created in a single transaction, sharing `Post.crossPostId` + an identical `createdAt` — comments/reactions/favorites stay scoped per group automatically, author `PATCH`/`DELETE` fan out to every sibling (`updateMany`/`deleteMany` on `{crossPostId, authorId}`), and `GET /` dedupes siblings within a page (plus a cursor-boundary check) so a member of several target groups sees the post once. Privacy: `crossPostId` is never serialized; only the author's responses carry `sharedWithGroups: [{id, name}]`, so non-authors can't tell a post was cross-posted. Linked-album assets in a multi-group post are copied server-side into `/uploads/` at create time (`services/media/copyAsset.ts`, via the providers' `readAsset()`) so every target group can see that one photo without any access to the source album; the source album's link must belong to one of the target groups. The `post.created` event carries all created posts and the notifications subscriber notifies each recipient once across the target groups. Admin content moderation stays per-group (deleting one sibling leaves the others). `type` is an open string (`UPDATE`/`MILESTONE`/`POLL`/`TRIP`, more later) resolved against the `PostTypeHandler` registry (`services/postTypes/`, a third extension-point directory mirroring `MediaProvider`); `POST` also takes handler-owned `typeData` (a poll's `{options:[{text}], closesAt?}`; a trip's `{title, destination?, startDate, endDate?, coverPhotoUrl?, travelerUserIds?}` — `title` 1-120 chars, `startDate`/`endDate` plain `YYYY-MM-DD`, `travelerUserIds` up to 20 other members, author implicit and stripped if included; each must be a current member of **every** target group, 400 `errors.tripTravelerNotMember` — a cross-posted trip stays one journal whose check-ins land in all sibling groups) — 400 `errors.unknownPostType`/`errors.invalidPostTypeData`, and 403 `errors.postTypeNotAllowed` if any target group's `allowedPostTypes` excludes the requested type (cross-post: **every** target group must allow it). List endpoints and `GET /:id` return `typeData` plus, for polls, an enriched `poll` object (`{options:[{id,text,voteCount,voters:[{id,name,avatarUrl}]}], totalVotes, myVoteOptionId, closesAt, closed}`) computed read-time from `PostInteraction` rows, and for trips, an enriched `trip` object (`{title, destination, startDate, endDate, coverPhotoUrl, travelers:[{id,name,avatarUrl}], closed, closedAt, dayNumber|null, durationDays|null, stopCount, photoCount, latestCheckin:{commentId,place,createdAt}|null, collagePhotoUrls[]}` — `closed` true once explicitly closed or `endDate`'s UTC day has fully elapsed) computed read-time from the trip's check-in `Comment` rows; `POST`/`PATCH` responses carry `typeData` but not `poll`/`trip` (same precedent as `people`). `POST /:postId/interactions` `{key, value}` is the generic per-type action endpoint — poll voting (`key:'vote'`, `value:{optionId}`; same option again unvotes, a different one switches; closed poll → 400 `errors.pollClosed`) and, for `TRIP` posts, three keys that on a cross-posted trip fan out to every sibling copy (the handler receives `crossPostId` — internal plumbing, never serialized): `checkin` `{place 1-80, text?<=2000, photoUrls?<=10 uploads-paths}` (author or a co-traveler; closed trip → 400 `errors.tripClosed`; before `startDate` → 400 `errors.tripNotStarted`; non-traveler → `errors.tripNotTraveler`; creates one `Comment` copy per sibling post in a single transaction, all sharing `metadata.checkinId` — but only in sibling groups the check-in author is still a member of at that moment), `close` (author-only, once, all siblings atomically — `errors.tripNotAuthor`/`errors.tripClosed`), `setTravelers` `{userIds[]}` (author-only, active-trip-only, replaces the traveler list wholesale on all siblings atomically, validated against **every** sibling group's membership → `errors.tripTravelerNotMember`) — returning the full shaped+enriched post; the route itself emits no domain event for an interaction (no vote notifications in v1), but a handler may emit one from inside its own `interact()` (a trip check-in emits one `comment.created` carrying every copy via `checkinTargets`, since check-ins are stored as `Comment` rows — see `comments.ts` row below) |
| `comments.ts`, `likes.ts`, `favorites.ts` | `/api` | `favorites.ts` adds `POST /posts/:postId/favorite` (toggle) and `GET /favorites` (the logged-in user's favorited posts, across all their groups). `GET /posts/:postId/comments` takes an optional `?assetUrl=` filter and `POST` an optional `assetUrl` body field to pin a comment to one photo/video (`Comment.assetUrl`) instead of the post as a whole; replies inherit the parent's `assetUrl`. `POST` also takes an optional `attachmentUrl` — a photo/video the commenter uploaded via `POST /api/uploads` as part of the comment itself (`Comment.attachmentUrl`), distinct from `assetUrl` — and `content` is then optional: the two are independently optional, but at least one is required, so a comment can be photo/video-only. `POST` also takes an optional `mentionedUserIds` array — the client resolves "@name" against the group member list, and the server re-validates each id is a current member of the post's group before notifying them (`mention` instead of `new_comment`, so they aren't double-notified). Comment responses also carry `metadata` (nullable `Json`, `Comment.metadata`) — `null` for every ordinary comment, and a handler-owned payload for a comment created *by* a post-type handler instead of this route (today, only a `TRIP` check-in: `{kind:'trip_checkin', checkinId, place, photoUrls}` — `checkinId` shared by every copy one check-in creates, one per cross-post sibling, present on single-group trips too); the public `POST` body schema never accepts a client-sent `metadata`, so `metadata.kind` is a trustworthy discriminator the notifications subscriber uses to route it to `trip_checkin` instead of `new_comment`/`mention`. `PATCH` and `DELETE /comments/:id` both fan out when the **author** edits or deletes a cross-posted trip's check-in (every sibling copy matching `checkinId` gets the same content update or is removed, mirroring author post edit/deletion); an admin deleting someone else's check-in stays per-group, like admin post moderation. `likes.ts`'s `POST /posts/:postId/like` and `POST /comments/:commentId/like` take an optional `{ type }` body (`ReactionType`: `LIKE`/`LOVE`/`HAHA`/`WOW`/`SAD`/`CARE`, default `LIKE`) — tapping the same reaction again removes it, tapping a different one switches it (still one reaction row per user per post/comment). `GET /posts/:postId/reactions` (`likes.ts`) returns every reactor plus which emoji they left (`{ items: [{ id, name, avatarUrl, type }] }`, newest first) — the full breakdown behind mobile's tap-the-avatars "who reacted with what" view, as opposed to `Post.recentReactors` (top 3, no `type`, embedded on every post response) |
| `chat.ts` | `/api/chat` | **Per-group chat** (off by default, toggled per-group by admins via `Group.chitchatEnabled`; 403 `errors.chitchatDisabled` if requested group has it disabled): `GET /groups/:groupId/messages?cursor=&take=` lists the chat, newest-first, cursor-paginated (default `take` 30, max 100); `POST /groups/:groupId/messages` creates a message (`{ content?: string, attachmentUrl?: string }`, at least one required — same attachment shape as comments, an `/uploads/` path); `DELETE /messages/:id` hard-deletes (author or admin only, 403 otherwise). `POST /groups/:groupId/read` marks the caller's read position (upserts a `ChatRead` row with `lastReadAt`). `GET /unread-counts` returns `{ [groupId]: number }` for every group the caller belongs to (groups with no unread messages don't appear). Message responses carry `{ id, groupId, authorId, author: {id, name, avatarUrl}, kind: 'USER' | 'SYSTEM_MILESTONE', content: string | null, attachmentUrl: string | null, refPostId: string | null, createdAt, editedAt, readBy: [{id, name, avatarUrl}] }` — `readBy` is derived read-time from `ChatRead` rows, not a per-message receipt table. New `NotifyType` value `new_chat_message` (push-only, no email) notifies every other group member except on `SYSTEM_MILESTONE` messages (the milestone post itself already triggered a `new_post` notification, avoiding double-notify). See [Data model](#data-model) for `ChatMessage` and `ChatRead` |
| `push-tokens.ts` | `/api/push-tokens` | `POST` registers, `DELETE` unregisters (called on mobile logout) |
| `api-tokens.ts` | `/api/api-tokens` | Self-service developer **personal access tokens** (`famlin_pat_...` bearer credentials for members' own scripts/integrations — documented for end users in `docs/developers/api.md`): `GET` lists the caller's tokens (never the secret), `POST` creates (plaintext returned exactly once; **session-authenticated requests only** — a PAT gets a 403, so a leaked token can't mint replacements and outlive revocation), `DELETE /:id` revokes. Verification lives in `services/apiTokens.ts` (SHA-256 hash lookup + expiry check) and is wired into `plugins/auth.ts`: `authenticate` branches on the `famlin_pat_` prefix and sets `request.authMethod` (`'session'` \| `'api-token'`), and `authenticateMediaRequest` accepts PATs for `/uploads/*`/Immich media. PATs are deliberately independent of `User.tokenVersion` (password reset keeps integrations running; revocation = row deletion, user deletion cascades). Managed from the web app's avatar menu (`web/src/components/ApiTokensModal.tsx`) — **web only by design, no mobile screen** |
| `notifications.ts` | `/api/notifications` | in-app notification history |
| `uploads.ts` | `/api/uploads` | direct photo upload to a Docker volume, served at `/uploads/` (auth-gated, see below); `GET /media-token` issues the token clients use to read it. Image uploads (`jpg`/`jpeg`/`png`/`webp`/`heic`/`heif`) are compressed server-side via `services/uploadVariants.ts`: the canonical `/uploads/<uuid>.jpg` path — the same URL every client, old or new, already requests — serves a resized display copy (`sharp`, 1920px max, q85), never the raw upload, so already-installed app versions benefit without an update; the true original is kept at `uploads/originals/<uuid>.<ext>` (never served — 404'd in `app.ts`'s `/uploads/` hook), and an opt-in `-thumbnail.jpg` (400px/q80) is generated for updated clients' grid tiles via `getUploadUrl(path, 'thumbnail')`. `.gif` uploads and undecodable files are stored/served as-is, unprocessed. Video uploads (`mp4`/`mov`/`m4v`/`webm`) are stored/served as-is too, but additionally get a best-effort `-thumbnail.jpg` poster frame (`generateVideoPoster` in `uploadVariants.ts`, ffmpeg's `thumbnail` filter, 400px) so clients can render video tiles as plain images — mobile's `MediaThumbnail` derives the poster URL (`getVideoPosterUrl` in `mobile/src/utils/media.ts`) and only mounts a real video player when the poster 404s (pre-poster uploads) or can't exist (local picker URIs); a poster failure never fails the upload. Applies to new uploads only — no backfill of historical uploads. HEIC/HEIF decode support requires `sharp` built against Alpine's `vips-heif` package (`backend/Dockerfile`'s `production` stage), since `sharp`'s own prebuilt binaries omit it; ffmpeg is installed in both the `base` (dev) and `production` stages for the video posters |
| `media.ts` | `/api/media` | **Read-only, member-facing, provider-generic** (backed by the `MediaProvider` registry, `services/media/registry.ts` — `immich` + `local` today): `GET /groups/:groupId/albums` (a group's linked albums across every media source, each carrying its `provider`; a link whose source is failing reports `assetCount: 0` rather than sinking the whole list), `GET /albums/:linkId/assets` (asset list with optional `?personId=` filter to show only assets of a mapped person; 404 if person isn't mapped, 400 if provider can't filter by person), `GET /groups/:groupId/photos` (merged photo timeline across linked albums + direct post uploads, cursor-paginated with optional `?personId=` filter; 403 non-member, 404 unmapped personId; backed by `services/media/photoTimeline.ts` + `services/media/personFilter.ts`; per-album asset listings are cached 60s and served stale-while-revalidate, and an album item that a post embeds via its media-proxy URL carries that post's `postId` + the verbatim stored URL as `postAssetUrl`, enabling the mobile viewer's like/comment/favorite bar on it), `GET /assets/:linkId/:assetId/:variantExt` (streams a rendition from whichever provider the link belongs to; `thumbnail`/`preview` are always `.jpg`, `original` matches the real asset type — see `parseMediaAssetPath()` in `types.ts`; `streamAsset()` implementations honor the client's `Range` header and produce/relay `206`/`Content-Range`, which native video players require to play mp4s at all — don't strip that, it breaks video on mobile only, since web `<video>` tolerates a full-body `200`). Gated by `authenticateMediaRequest` (`plugins/auth.ts`), the same dual session-token-or-media-token check `/uploads/*` uses, since it's loaded as an `<Image>`/`<Video>` `uri` — and by the provider's `isAssetInAlbum()` (a required part of the `MediaProvider` contract, not an optimization; Immich caches it 60s, local stats the file), which re-checks the requested asset id actually belongs to the linked album rather than trusting the linkId→group check alone. Linking/unlinking albums is an admin mutation in `admin.ts` (`/api/admin/groups/:id/media-albums`, `/api/admin/media-albums/:id`, per-provider catalog `/api/admin/media/:provider/albums`, connection tests `/api/admin/immich/test` + `/api/admin/media/local/test`). New: `GET /api/media/people?groupId=` (lists people mapped to this group, deduplicated per (provider, label) pair: `[{id, provider, label, userId?}]`, 403 if non-member; same-label mappings appear as one entry with a representative id) — admin mutations for people mapping are in `admin.ts` |
| `immich.ts` | `/api/immich` | **Legacy compat, Immich-only**: the same three member endpoints as `media.ts` but scoped to `provider: 'immich'` links and generating `/api/immich/assets/...` URLs. Kept because that URL shape is stored in existing posts' `uploadedAssetUrls` and still generated by pre-built mobile clients — new client code should use `/api/media`. Both files delegate to the same `immichProvider` (`services/media/immich.ts`) |
| `invites.ts` | `/api/invites` | **Public**: invite preview (`GET /:token`), self-service registration (`POST /:token/register`), join-for-logged-in-user (`POST /:token/accept`, authenticated). Creation/list/revoke are admin mutations in `admin.ts` instead (`/api/admin/groups/:id/invites`, `/api/admin/invites/:id`) |
| `invite-landing.ts` | `/invite/:token` (no prefix) | **Public**: server-rendered HTML the invite link opens; hands off to the app via `famlin://invite/:token?server=...`. Matches the app's design (logo, teal palette) and is translated into the server's `defaultLanguage` setting (no per-visitor Accept-Language negotiation, since there's no signed-in user yet) |
| `landing.ts` | `/` (no prefix) | **Fallback only**: server-rendered HTML landing page (logo + welcome message, same shell as the invite landing page via `utils/html-page.ts`), registered only when `dist/web` doesn't exist. Normally the `web/` SPA is served at `/` instead, with an SPA `index.html` not-found fallback for GET routes outside `/api/*`, `/uploads/*`, and `/admin*` |

Services live in `backend/src/services/` (`settings.ts`, `notifications.ts`, `invites.ts`, `users.ts`, `groups.ts`, `posts.ts`, `comments.ts`, `reactions.ts`, `onThisDay.ts`, `pagination.ts`, `apiTokens.ts`, plus three extension-point directories: `media/` — the `MediaProvider` interface/registry with the `immich` and `local` providers, see the `media.ts` route row above — `notificationChannels/` — the `NotificationChannel` interface with the `push` and `email` channels, see the notifications bullet in Conventions; `sendPush()` in `notificationChannels/push.ts` is the single write path for `PushDeliveryLog` (one row per Expo send attempt, organic or admin-triggered — see `admin.ts`'s `retrigger-push`/`push-log` routes above) — and `postTypes/` — the `PostTypeHandler` interface/registry (`registry.ts`) with the `update`/`milestone`/`poll`/`trip` handlers, modeled directly on the `MediaProvider` pattern: one file per post type (`update.ts`/`milestone.ts`/`poll.ts`/`trip.ts`), registered by id in a static map, never renamed once a deployment has posts of that type. A handler's optional hooks are `typeDataSchema` (zod, validates client-sent `typeData` at create time), `validateCreate`/`transformCreate` (semantic checks + persisted-shape transform — given `authorId`/`targetGroupIds`/`{authorId}` respectively; e.g. `poll.ts` assigning each option a `crypto.randomUUID()` id, or `trip.ts` rejecting cross-posting and stripping the author from `travelerUserIds`), `interact` (handles `POST /api/posts/:postId/interactions` for that type, given `post:{id,typeData,authorId,groupId,groupName,crossPostId}` (`crossPostId` lets a handler fan out to siblings; internal plumbing, never serialized) — poll voting: `key:'vote'`, unvote/switch semantics; `trip.ts`: `checkin`/`close`/`setTravelers`, throws `PostTypeError` for a closed poll/trip or unknown option/non-traveler), and `enrichPosts` (batch-attaches computed fields — the poll's live tallies, the trip's aggregates — to a page of same-type posts in one query, called from `shapePostsWithPeople` in `services/posts.ts` alongside the `people` enrichment so every list endpoint and `GET /:id` gets it for free). `trip.ts` is the one handler whose persisted `typeData` isn't fully immutable after creation — `closedAt`/`closedByUserId`/`travelerUserIds` are further written by its own `interact()` (`close`/`setTravelers`, via `updateMany` on `{crossPostId, authorId}` so every cross-post sibling stays identical), a documented exception since closing a trip or changing its travelers is part of the type's lifecycle rather than a post edit; a cross-posted trip stays **one** journal — travelers must be members of **every** target group (create and `setTravelers` both check), and its `checkin` interaction creates one `Comment` copy per sibling post in a single transaction (`metadata:{kind:'trip_checkin', checkinId, place, photoUrls}`, `checkinId` shared across copies, skipping sibling groups the check-in author is no longer a member of — see the `comments.ts` row above; all three interactions take `SELECT ... FOR UPDATE` row locks on the sibling rows and re-read `typeData` inside the transaction, so concurrent `close`/`setTravelers`/`checkin` calls can't clobber each other's typeData writes) and emits one `comment.created` carrying every copy via `checkinTargets`, so the notifications subscriber routes it to `trip_checkin` and notifies a member of several sibling groups exactly once (the `post.created` dedupe pattern). `resolveAllowedPostTypes(group)` also lives here — see `Group.allowedPostTypes` below). `src/events.ts` is the typed in-process **domain event bus** (`post.created`, `comment.created`, `reaction.added`): routes emit facts after their write commits; subscribers in `src/subscribers/` (registered idempotently by `buildApp()`, so tests get the same wiring) decide what the facts mean. Every handler runs fire-and-forget and error-trapped — a broken subscriber can never fail or slow the emitting request (pinned by `tests/events.test.ts`). The notifications subscriber (`src/subscribers/notifications.ts`) owns all "who gets told what" policy (thread participants, mention validation, reaction targets) — routes know nothing about notifications; don't re-add direct `notifyGroup`/`notifyUsers` calls to routes. The admin SPA is served at `/admin` with `index.html` fallback for client-side routing. `src/jobs/onThisDay.ts` runs once a day and `src/jobs/newAssets.ts` runs hourly (minute 15, both via `node-cron`, scheduled in `server.ts` — not `app.ts`, so the test suite building the app via `buildApp()` never triggers jobs): onThisDay notifies each group about posts made on that day in a prior year, while newAssets diffs linked albums against their provider and either notifies the group (MANUAL mode) or auto-creates a post (AUTO mode) when new assets appear.

**List endpoints are cursor-paginated.** `GET /api/posts`, `GET /api/posts/search`, `GET /api/favorites`, and the admin `/users`/`/content/posts`/`/content/comments` endpoints take `?cursor=`/`?take=` (default 30, max 100) and return `{ items, nextCursor }` instead of a bare array — use `services/pagination.ts`'s `paginationArgs`/`paginate` for any new list endpoint rather than returning everything unbounded. `GET /api/posts/on-this-day` doesn't paginate (expected result set is small: at most a handful of years' worth of same-day posts).

**Data model** (`backend/prisma/schema.prisma`): `User`, `Setting`, `Group` (`allowedPostTypes: String[]`, `@default([])`, restricts which post types members may create in that group — empty means every registered type, including ones added after the group was created, same convention as `allowedEmails`; resolved via `resolveAllowedPostTypes()` in `services/postTypes/registry.ts`, see the `groups.ts`/`admin.ts` rows above; `chitchatEnabled: Boolean`, `@default(false)`, toggles group chat on/off per-group, admin-controlled via group create/update endpoints in `admin.ts`), `GroupMember` (M2M join, `@@unique([groupId, userId])`), `Post` (belongs to exactly one `Group`; nullable indexed `crossPostId` links the sibling rows of one cross-posted message — see the `posts.ts` row above; `type: String` is an open discriminator, `@default("UPDATE")` — was a closed `PostType` enum — resolved against the `PostTypeHandler` registry (`UPDATE`/`MILESTONE`/`POLL`/`TRIP` today, never rename a persisted id); `typeData: Json?` is handler-owned, type-specific config, `null` for `UPDATE`/`MILESTONE`, a poll's `{options:[{id,text}], closesAt}` for `POLL`, a trip's `{title, destination, startDate, endDate, coverPhotoUrl, travelerUserIds, closedAt, closedByUserId}` for `TRIP` — validated/transformed only at creation and immutable afterward, except a `TRIP`'s `closedAt`/`closedByUserId`/`travelerUserIds`, which its own handler writes later via the `close`/`setTravelers` interactions — atomically across all cross-post siblings, which always share identical typeData), `PostInteraction` (generic per-user, per-post state for a type's own features — a poll vote, `key:'vote'`/`value:{optionId}`, more later; `@@unique([postId, userId, key])` caps it at one row per user per post per key, indexed on `[postId, key]`; a `TRIP`'s check-ins deliberately don't live here, see `Comment.metadata` below), `Comment` (`attachmentUrl` optionally holds a photo/video the commenter uploaded as part of the comment itself, distinct from `assetUrl`'s pin to an existing post asset; `metadata: Json?` is a handler-owned payload for a comment created *by* a post-type handler rather than the plain comments route — `null` for every ordinary comment, `{kind:'trip_checkin', checkinId, place, photoUrls}` for a `TRIP` check-in (`services/postTypes/trip.ts`; `checkinId` is shared by every copy one check-in creates across cross-post siblings, single-group trips included, and drives the author's delete fan-out in `routes/comments.ts`) — the public comment-create route never accepts a client-sent `metadata`, so `metadata.kind` is a trustworthy discriminator), `Like` (a reaction — `type: ReactionType` is `LIKE`/`LOVE`/`HAHA`/`WOW`/`SAD`/`CARE`, one row per user per post/comment via `@@unique([postId, userId])`/`@@unique([commentId, userId])`; post responses also expose `recentReactors` — the three newest reactors' `{ id, name, avatarUrl }`, via `shapePost` in `services/posts.ts` — for the mobile app's overlapping reaction avatars); post responses also carry `people` — an array of admin-mapped media people recognized in that post's photos, computed read-time in `services/media/personTags.ts` by querying each linked album's asset-to-people index (Immich metadata search + per-asset fallback, 10-minute cache) and merging tagged assets with the same person-label into one chip, with fail-soft semantics (provider outage → empty array, never a failed request). `Favorite` (a user's bookmarked posts, `@@unique([postId, userId])`, mirrors `Like`), `ChatMessage` (belongs to a `Group` and an author `User`; `kind: String` is an open discriminator — `'USER'` for member-sent messages, `'SYSTEM_MILESTONE'` for auto-generated messages when a milestone post is created in a group with chitchat on, `refPostId` nullable link to the milestone `Post`; hard-deleted like posts/comments), `ChatRead` (one row per `(groupId, userId)` pair, tracks `lastReadAt` — upserted by `POST /api/chat/groups/:groupId/read`, used to compute `readBy` on message responses — no per-message receipt table), `PushToken`, `PushDeliveryLog` (one row per push-send attempt — organic or an admin's manual "resend push" — written by `sendPush()` in `services/notificationChannels/push.ts`; `postId`/`triggeredByAdminId` are nullable `SetNull` relations so deleting the post/admin later doesn't erase the log, `notifyType` mirrors `Notification.type`'s open-string convention, `recipientCount`/`tokenCount`/`successCount`/`failureCount` are the per-attempt tallies surfaced by the admin UI's push-log page), `ApiToken` (a developer personal access token — stores only a SHA-256 `tokenHash` of the `famlin_pat_...` secret plus a display `tokenPreview`, optional `expiresAt`, throttled `lastUsedAt`; see `api-tokens.ts` above), `Notification`, `Invite`, `MediaAlbumLink` (links one external media album to one group: `provider` is the registry id — `'immich'`/`'local'`, never rename these once deployments have linked albums — and `externalAlbumId` is that provider's album identifier (an Immich uuid, a local folder name), `@@unique([groupId, provider, externalAlbumId])`; `newAssetMode` controls how new assets on this album are surfaced (`OFF` = never, `MANUAL` = a notification reusing the new-post preference columns, `AUTO` = an auto-created post); `newAssetsCheckedAt` is the last time the new-assets job ran for this link, used to diff against the provider. The proxy routes additionally re-check the requested asset actually belongs to that album, see `media.ts` above. Born as `ImmichAlbumLink` — renamed + backfilled by the `20260709100000_generalize_media_album_link` migration), `MediaPersonLink` (maps a provider's face/person identity to a display label and optional user: `provider`/`externalPersonId` unique, `userId` nullable; only admin-mapped people are visible to members; enables per-person asset filtering). `Post`, `Comment`, and `User` are all hard-deleted — `DELETE /api/posts/:id`, `DELETE /api/comments/:id`, and `DELETE /api/admin/users/:id` permanently remove the row via `prisma.*.delete()`, with cascades doing the rest (deleting a post removes its comments/likes/favorites/notifications; deleting a user removes their posts/comments/likes/favorites/etc. too). There is no restore. `User.tokenVersion` is embedded in every JWT and checked on each request, so bumping it (password change/reset) invalidates every previously-issued token immediately. `User.pushOnChitchat` (Boolean, `@default(true)`) is a notification preference for chat messages (push-only by design, no email), updatable alongside the existing `pushOnNewPost`/`pushOnNewComment`/`pushOnNewLike` preferences via `PATCH /api/auth/me`. `Notification.type` is a plain string (not a DB enum) — the closed set of values (including `mention`, `on_this_day`, `new_chat_message`, and `trip_checkin`) lives in `NotifyType` in `services/notificationChannels/types.ts` (re-exported from `services/notifications.ts`); `new_chat_message` (push-only, no email, reuses `pushOnChitchat` column) fires on every other group member except for `SYSTEM_MILESTONE` chat messages (the milestone post itself already triggered `new_post`, avoiding double-notify); `trip_checkin` (push-only, no email, reuses `pushOnNewPost` column) fires on every other group member when a `TRIP` post gets a check-in, with same-day check-ins from the same author bundled into one notification instead of the normal `new_comment`/`mention` path a plain comment would take — on a cross-posted trip the event's `checkinTargets` lets the subscriber notify a member of several sibling groups exactly once (the `post.created` dedupe pattern); others reuse the existing email/push columns.

**Mobile** (`mobile/src/`): screens in `screens/`, navigation in `navigation/` + `App.tsx`, Zustand store in `stores/`, API layer in `api/` (axios; base URL is user-entered at login, not hardcoded), React Query for fetching/caching.

## Conventions & gotchas

- **Group membership is the core authorization rule.** Every posts query must filter on groups the logged-in user belongs to — a user must never see posts from a group they aren't a member of. Group/member mutations require `isAdmin`.
- **Group mutations live only in `admin.ts`.** `groups.ts` is deliberately read-only — do not re-add create/update/delete or member-management routes there.
- **`requireAdmin` returns a boolean and sends the 403 itself.** Callers **must** `return` when it returns `true`, otherwise the handler keeps running after the response is sent.
- **Runtime config is in the DB, not `.env`.** OIDC issuer/client ID/scopes, allowed emails, SMTP, Immich server URL/API key, the local-media root path (`localMediaPath`), and the server's `defaultLanguage`/`appStoreUrl`/`playStoreUrl` live in the `Setting` table (managed via the admin UI's "General"/"Media integrations" sections), read through `getAllSettings()` (`services/settings.ts`), which caches for 10s and is invalidated on write. Each media source is one server-level connection shared by every group — `getImmichSettings()`/`getLocalMediaSettings()` report whether it's `configured`; which albums are visible to which group is a separate per-group `MediaAlbumLink` row, not part of `Setting`. `defaultLanguage` is used both for server-rendered pages with no signed-in user (the `/invite/:token` landing page) and for notification push/email text, since recipients don't have a stored per-user language preference — see `getT`/`SUPPORTED_LANGUAGES` in `src/i18n/index.ts`. `appStoreUrl`/`playStoreUrl` are optional; when set (or defaulted), the invite landing page shows download buttons. `appStoreUrl` is blank by default (no single iOS listing — each deployment distributes its own build). `playStoreUrl` defaults to the official pre-built Android app (`https://play.google.com/store/apps/details?id=be.xeweb.famlin`, package `be.xeweb.famlin` per `mobile/app.json`) whenever it's never been explicitly saved — an admin can still clear it to hide the button, which is preserved as a real blank rather than falling back to the default (see the `map.has()` check in `loadAllSettings()`, `services/settings.ts`). Only `DATABASE_URL`, `PORT`, `NODE_ENV`, `JWT_SECRET`, and `TRUST_PROXY` are startup env vars — `TRUST_PROXY` (default `false`) must be `true` when (and only when) Famlin sits behind a reverse proxy that sets `X-Forwarded-Proto`/`X-Forwarded-Host`, otherwise those headers are ignored so a directly-exposed server can't have its origin spoofed.
- **OIDC login works with any provider** (Google, Microsoft, Authentik, Keycloak, Auth0, ...) via standard discovery + JWKS verification (`jose`, see `plugins/auth.ts` `verifyOidcToken`/`getDiscovery`). By default, mobile and admin both do a public-client Authorization Code + PKCE flow directly against the provider (no client secret) and hand the resulting ID token to `POST /api/auth/oidc`; `GET /api/auth/oidc-config` is the public endpoint clients use to discover the issuer's endpoints (plus `usesClientSecret`/`mobileCallbackUrl`, see below). It respects `allowedEmails` — new accounts are only provisioned if the email is whitelisted (empty list = allow all); see `services/settings.ts` `isEmailAllowed`. All three OIDC entry points share one user-provisioning/token-issuing function, `completeOidcLogin()` in `services/oidcLogin.ts`.
- **Google needs the optional `oidcClientSecret` setting, not the plain PKCE flow above.** Google's token endpoint rejects a secretless exchange, and its native/"iOS"/"Android" OAuth client types force `redirect_uri` onto a Google-generated `com.googleusercontent.apps.<id>` scheme rather than the app's own `famlin://` — a hard blocker since Famlin ships one shared pre-built Android app (`be.xeweb.famlin`) across many independent deployments, each with a different Google client and therefore a different mandated scheme that can't be baked into that shared binary. Setting `oidcClientSecret` switches both surfaces to a server-mediated exchange instead, so Google can be configured as one ordinary **Web application** client: the admin UI keeps its existing `https://.../admin/` redirect but POSTs the code to `POST /api/auth/oidc/exchange` instead of exchanging it itself; the mobile app authorizes against `mobileCallbackUrl` (an HTTPS URL under the deployment's own domain, from `GET /oidc-config`), and `GET /api/auth/oidc/mobile-callback` finishes the exchange server-side and 302s to `famlin://oidc-callback?handoff=<code>` — a real redirect_uri scheme change would break the deep link contract, so the token itself is never put in that URL's query string (would leak into access logs); the app redeems the one-time `handoff` code via `POST /api/auth/oidc/mobile-handoff` (`services/oidcHandoff.ts`, in-memory, single-use, 2-minute TTL — see `utils/oidcLogin.ts` in the mobile app).
- **i18n is mandatory — no hardcoded UI strings.** Use `const { t } = useTranslation()` and keys. Locales: `mobile/src/i18n/locales/{en,nl}.json` and `backend/admin/src/i18n/locales/{en,nl}.json`; `en.json` is the source, `nl.json` the translation. Backend error messages use `getT(request)('errors.…')`. Adding a UI string means adding both `en` and `nl` keys. Weblate component type: **i18next JSON**; `fallbackLng` is `en`.
- **Backend is ESM + `NodeNext`.** JSON imports need the attribute: `import x from './x.json' with { type: 'json' };`.
- **Add/bump mobile native dependencies with `npx expo install <package>`, never by hand-editing `mobile/package.json`.** Expo precompiles native frameworks (RNReanimated etc.) against the exact versions in `expo/bundledNativeModules.json`; a manually pinned different version builds from source instead, and the resulting mixed precompiled/source linkage crashes the released app at launch (dyld "Library not loaded") while dev builds, `tsc`, and Jest all stay green — the worklets 0.10.2 incident. `npx expo install --check` guards this in both `ci.yml` (Mobile job) and `mobile-build.yml` (before the store build).
- **Removing a group member keeps their existing posts/comments visible** to the remaining members (deliberate MVP choice).
- **Posts, comments, and users are all hard-deleted — there is no soft-delete anywhere.** `DELETE /api/posts/:id`/`DELETE /api/comments/:id` call `prisma.post.delete`/`prisma.comment.delete` directly, and `DELETE /api/admin/users/:id` calls `prisma.user.delete`; nothing filters on a `deletedAt` column because that column doesn't exist. Deleting a user cascades to remove their posts, comments, likes, favorites, and notifications immediately (`onDelete: Cascade` on `Post.author`/`Comment.author` etc. in `schema.prisma`) — this is intentional, don't add back a `deletedAt` column or a restore endpoint to soften it. The last remaining admin can't be deleted or demoted (`isLastAdmin` in `routes/admin.ts`) — keep that check if you touch those routes. `backend/src/__tests__/authorization.test.ts` has regression tests confirming a deleted post/comment/user is fully gone (404s/401s, not just hidden).
- **There is no default/seeded admin account.** A fresh install has zero users; `GET /api/auth/setup-status` reports `needsSetup: true` until the first user exists, and the admin UI shows a one-time setup screen (`SetupPage.tsx`) instead of the login form in that state. `POST /api/auth/setup` provisions that first account as an admin and is guarded by a Postgres advisory lock (`SETUP_ADVISORY_LOCK_KEY` in `routes/auth.ts`) so two concurrent requests can't both pass the "no users yet" check — it 409s once a user exists. `prisma/seed.ts` (`db:seed`) is a **local dev/test fixture script only** (fake family + `admin@example.com`/`test123456`), never run automatically by the Docker image or `prisma migrate deploy` — don't treat it as production onboarding.
- **Invite links are their own authorization and deliberately bypass `allowedEmails`.** A valid, unexpired, unused `Invite` (generated per-group from the admin UI) lets `POST /api/auth/oidc` / `POST /api/auth/login` (via an optional `inviteToken` field) and `POST /api/invites/:token/register` provision or join a user even when the email isn't whitelisted — see `verifyOidcToken(idToken, { allowUnlisted })` in `plugins/auth.ts` and `consumeInvite`/`getValidInvite` in `services/invites.ts`. Don't add new email-allowlist checks upstream of these paths without accounting for the invite override. `consumeInvite` runs as a transaction with an atomic conditional update so two concurrent redemptions of the same single-use invite can't both succeed.
- **`/uploads/*` requires auth — session token or media token.** See [Photos and uploads](docs/developers/architecture.md#photos-and-uploads) in the architecture doc. `uploadedAssetUrls`/`assetUrl` are validated (zod regex in `types.ts`) against the exact path shape the upload route writes — don't loosen that to a bare `z.string()`, it's what stops a post from linking to an arbitrary external URL.
- **Don't leak `err.message` to the client.** The global error handler (`app.ts`) turns a `ZodError` into a translated `400` and anything else into a generic translated `500`; a route's own `catch` block should only special-case errors it needs to give a specific, translated, non-sensitive message for (see the `OidcError`/`P2025` handling in `routes/auth.ts` as the pattern) — let everything else propagate to the global handler instead of echoing `err.message`.
- **Mark breaking changes in commit messages — release notes depend on it.** Commits follow Conventional Commits, and any change that would require action from an existing deployment on upgrade (incompatible API/response change, newly required env var, behavior-altering default) must carry a `!` after the type (`feat!: ...`) or a `BREAKING CHANGE: <what an admin must do>` footer. release-please builds the release's "⚠ BREAKING CHANGES" changelog section and the version bump from that marker, and the AI release-notes rewrite in `release-please.yml` only warns admins about marked commits — an unmarked breaking change ships silently. The `api-breaking-changes` job in `ci.yml` enforces the API half: it runs oasdiff on `docs/openapi/famlin.yaml` and fails a PR whose spec change is breaking with no marked commit/PR title; non-API breakage (env vars, deploy steps) relies on the commit author — that includes you — marking it.

## Deploy

Docker Compose (`docker-compose.yml`): `famlin-db` (Postgres, persistent volume, **not** published to the host — reached only over the compose network) + `famlin-backend` (Node 20, runs as the non-root `node` user), pulled by default as the pre-built `ghcr.io/timvanonckelen/famlin:${FAMLIN_VERSION:-latest}` image rather than built locally. Requires `JWT_SECRET` and `POSTGRES_PASSWORD` in `.env` (see `.env.example`); `docker compose up` fails fast with a clear error if either is unset. First run: open `/admin` — a fresh database has no users, so it shows a one-time setup screen to create the first (admin) account — then configure OIDC/SSO, allowed emails, and SMTP there. Keep the Postgres volume in the existing NAS backup scope. `docker-compose.override.yml` (local dev only, auto-merged by plain `docker compose up`) is what actually builds from `backend/Dockerfile` for hot-reload — production deploys use `docker-compose.yml` alone and never build from source unless deliberately switched over (see `docs/docs/maintenance.md#building-from-source-instead`).

**`backend/Dockerfile` builds from the repo root, not `backend/`** — the image bundles the member-facing web app (`web/` + `packages/api-client`, both outside `backend/`) in a dedicated `webbuild` stage (root workspace `npm ci` + `vite build` → copied to `dist/web`), alongside the backend + admin build. Both `docker-compose.override.yml` and `docker-publish.yml` therefore use `context: .` with `dockerfile/file: backend/Dockerfile`; the root `.dockerignore` keeps the context lean. In local dev the container doesn't build web itself (the root workspace isn't mounted in) — instead the overlay bind-mounts `./backend/dist/web` through the anonymous `/app/dist` volume, so a **host-side** `npm run build:web` (or `npm run watch:web` for rebuild-on-change) makes the dev container serve the web app at `http://localhost:3000/`; restart the backend after the *first* build so it detects `dist/web/index.html` (`app.ts` checks for the file, not the directory, since the bind mount creates an empty dir). `npm run watch:web` also auto-refreshes the browser tab: it sets `LIVERELOAD=1`, which `web/vite.config.ts`'s `watchLivereload()` plugin uses to start a `livereload` server watching `outDir` and inject its client snippet into the built `index.html` — never set by plain `vite build`, so the real production bundle in the Docker image never carries it. Alternatively `npm run dev:web` runs the Vite dev server on 5174 with its own HMR, proxying `/api` to the container.

**`.github/workflows/docker-publish.yml`** builds and pushes `backend/Dockerfile` to `ghcr.io/timvanonckelen/famlin`, tagged with the release semver plus a rolling `latest`. It's a **reusable workflow** (`workflow_call` with an optional `tag_name` input) plus a manual `workflow_dispatch` entry point — it is **not** triggered by `on: release`, because release-please creates the release with the default `GITHUB_TOKEN` and GitHub suppresses workflow-triggered-workflow events. Instead, `.github/workflows/release-please.yml` cuts releases from conventional commits on `main` and, in the same run, chains a `publish-docker` job (gated on `release_created`) that calls `docker-publish.yml` with the new tag — so the image build needs no manual trigger once a release lands.

**Website & docs** deploy separately to GitHub Pages via `.github/workflows/pages.yml`, triggered on pushes to `website/**` or `docs/**`: it builds `docs/` with Docusaurus, assembles `website/` (root) + `docs/build` (under `/docs`) into one artifact, and publishes it to `famlin.app` (custom domain via `website/CNAME`).
